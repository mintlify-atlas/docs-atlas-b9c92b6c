---
title: "Query loop: the heartbeat of an agent system"
description: How Claude Code's continuous execution loop manages stateful multi-turn execution, recovery, and context budgeting
---

The easiest mistake when working with a code-writing model is imagining it as an upgraded Q&A API: user sends one message, model returns one output, task done. That mental model is not baseless — many model products work exactly this way. But once the system starts calling tools, spanning turns, handling interrupts, persisting state, retrying failures, and compacting context, one-shot Q&A understanding collapses.

Claude Code does not make that mistake. Structurally, it explicitly admits that agents depend on continuous, stateful execution. The center of that execution is the query loop.

<Note>
Whether a system deserves to be called an agent is often determined less by how it talks and more by whether it still knows what it is doing several turns later. The query loop is what makes that continuity possible.
</Note>

## The execution cycle

<Steps>
  <Step title="Input governance before model call">
    Before entering the model stage, runtime performs a long preparation sequence. In `queryLoop()` at `src/query.ts:241`, this includes:

    - Start memory prefetch (`src/query.ts:297`)
    - Prefetch skill discovery (`src/query.ts:323`)
    - Slice valid messages after compact boundary (`src/query.ts:365`)
    - Apply tool result budget (`src/query.ts:369`)
    - Perform history snip (`src/query.ts:396`)
    - Perform microcompact (`src/query.ts:412`)
    - Perform context collapse (`src/query.ts:428`)
    - Attempt autocompact last (`src/query.ts:453`)

    This sequence is an architectural statement. Claude Code puts context governance before model reasoning. It does not delegate "turn chaos into order" to the model — runtime governs first, then passes cleaner inputs through.
  </Step>
  <Step title="Model invocation via streaming">
    After governance steps, Claude Code enters model invocation around `src/query.ts:652`. Output is consumed with `for await` streaming, not as one synchronous complete response.

    Model output here is an event stream, not just final prose. Events may contain assistant text, `tool_use` blocks, usage updates, stop reasons, or API errors.

    After `src/query.ts:826`, the system stores assistant messages, extracts `tool_use` blocks, decides follow-up needs, and can dispatch tools to `StreamingToolExecutor` while streaming is still in progress.

    <Note>
    Streaming is not just about seeing words sooner. It lets runtime schedule next actions before model output fully ends. Once output becomes an event stream, architecture is no longer plain request-response — it becomes a drive-schedule-feedback process.
    </Note>
  </Step>
  <Step title="Tool execution and state update">
    Tool calls extracted from the stream are dispatched and executed. The results feed back into the loop's mutable state, assembled at `src/query.ts:268` as a unified `State` object:

    - `messages`
    - `toolUseContext`
    - `autoCompactTracking`
    - `maxOutputTokensRecoveryCount`
    - `hasAttemptedReactiveCompact`
    - `pendingToolUseSummary`
    - `stopHookActive`
    - `turnCount`
    - `transition`

    Claude Code does not scatter recovery, compaction, budget, hooks, and turn counting across ad hoc local booleans. It treats them as one basis for how the next turn continues after this turn ends.
  </Step>
  <Step title="Interrupt handling">
    After `src/query.ts:1011`, streaming abort is handled. If `streamingToolExecutor` is active, remaining results must be consumed and synthetic tool results generated, so issued `tool_use` blocks are not left without paired results. Otherwise `yieldMissingToolResultBlocks()` fills missing interruption records.

    Once the system has committed to execution externally, interruption still requires ledger closure. External systems, UI, and transcripts need a consistent causal chain. Even "interrupted" must be a complete record.
  </Step>
  <Step title="Recovery branches">
    Recovery in Claude Code is layered, not simplistic retry. Two common cases:

    **Prompt too long**: After `src/query.ts:1065`, the system checks whether the last assistant message is a withheld "prompt too long." If yes, it first tries to flush staged context collapse (`:1086` to `:1116`); if still insufficient, it enters reactive compact (`:1119` to `:1166`). Recovery proceeds from lower cost and lower destructiveness upward.

    **Max output tokens**: After `src/query.ts:1185`, runtime first tries raising the token cap; if still insufficient, it appends a meta message telling the model to continue exactly from the truncation point — not apologize, recap, or write polite filler.

    Recovery is part of the main runtime path, not post-failure etiquette.
  </Step>
  <Step title="Stop condition evaluation">
    Claude Code query loop distinguishes at least:

    - Streaming completed with `tool_use`, requiring follow-up
    - No `tool_use`, entering stop hooks and further checks
    - User interruption
    - Prompt-too-long recovery branch
    - Max-output-tokens recovery branch
    - Stop-hook blockage requiring loop re-entry
    - API errors returning directly

    From `src/query.ts:1062` through `:1305`, the stop-hooks section (`:1267` to `:1305`) explicitly prevents dead loops such as "still too long after compact, blocked by hook, compact again forever." Retry itself needs governance. Runtime must know why it is retrying, what it has already tried, and what patterns lead to infinity.
  </Step>
</Steps>

## State is core business, not baggage

Many systems initially treat state as a burden and statelessness as elegance. For agent systems, this preference has limited value. Once an agent enters real workflows, state emerges naturally. Ignoring it does not remove it — it only makes it return in less manageable form.

In `src/query.ts:203` to `:217`, mutable query-loop state is clearly defined as the `State` object listed above. These fields are updated as a whole through continue branches, not scattered across ad hoc locals.

<Note>
The key difference between mature agent systems and disposable scripts: scripts ask only whether this step finished. Agent systems must also ask whether the next step can continue from the state left by a failed step.
</Note>

## Query() vs queryLoop()

In `query()` at `src/query.ts:219` and `queryLoop()` at `src/query.ts:241`, the former is shell and the latter is core. `queryLoop()` is not a model call wrapped in `try/catch`. It maintains cross-iteration state, performs pre-governance steps, enters model streaming stage, then decides whether to execute tools, recover, compact, continue next turn, or terminate.

The model call is one phase of the loop, not the loop itself. Model call is one contraction inside the heartbeat. What keeps the system alive is the full cycle: input intake, stream consumption, tool scheduling, failure recovery, and turn continuation.

## QueryEngine: conversation lifecycle ownership

`QueryEngine` makes the loop's scope explicit.

At `src/QueryEngine.ts:176`, source states:

> QueryEngine owns the query lifecycle and session state for a conversation.

At `src/QueryEngine.ts:180`, one `QueryEngine` maps to one conversation, and each `submitMessage()` opens a new turn inside that conversation while preserving state.

After `src/QueryEngine.ts:675`, QueryEngine hands prepared `messages`, `systemPrompt`, `userContext`, `systemContext`, and `toolUseContext` to `query()`, then writes assistant/user/compact-boundary messages back into the transcript.

This means query loop is the actual execution center of conversation runtime. Outer UI, SDK, and session persistence all orbit it. To understand Claude Code you cannot stop at listing tools or reading prompt text. You must inspect how this loop turns those constraints into continuous behavior.

## The principle

The core capability of an agent system is maintaining a recoverable execution loop.

A mature agent heartbeat must satisfy at least:

- Explicit cross-turn state
- Active input governance rather than passive input consumption
- Stream-based model output handling
- Ledger closure after interrupt
- Clear distinction among completion, failure, recovery, and continuation

Without these structures, systems may still produce attractive demos, but they are closer to staged performances than runtimes. Performances have value, but they do not replace order.
